# QQ scan-only server route

Goal: the end user only scans the QQ client login QR once. After that the server opens QQ Classic Farm, captures the one-time farm `code`, and the bot takes over the farm WebSocket.

## Judgement

The runnable QQ farm automation projects all split into two layers:

1. Get the one-time farm `code` generated by the QQ mini-app runtime.
2. Use that `code` to connect to `wss://gate-obt.nqf.qq.com/prod/ws` and speak the farm Protobuf protocol.

This repo already implements layer 2. The broken part is layer 1. The old `q.qq.com/ide/login` QR exchange returns errors like `-3000`, so a plain web/OAuth QQ scan cannot produce the farm code anymore.

The executable scan-only route is therefore:

1. Run a real QQ client/QQ mini-app runtime on the server.
2. User scans the QQ client login QR once.
3. Server opens QQ Classic Farm in that logged-in runtime.
4. Patch the local QQ Farm mini-app `game.js` to wrap `qq.connectSocket` / `wx.connectSocket`.
5. When the real mini-app connects to `gate-obt.nqf.qq.com/prod/ws?...code=...`, the wrapper sends that URL's code to the bot's capture endpoint.
6. `/api/code-capture` creates or updates the account and starts the bot worker.

## Files

- `tools/sniff9988.py`: receives captured code and forwards it to the panel.
- `tools/patch-qq-farm-code-capture.js`: patches QQ Farm `game.js` in the QQ runtime cache.
- `tools/watch-qq-farm-code-capture.js`: waits for the QQ Farm cache and patches it automatically.
- `tools/start-local.ps1`: starts the local panel, capture service, and patch watcher.
- `tools/install-linux-service.sh`: installs panel and capture services.

## Server prerequisites

The server must be able to run a graphical QQ client. On a headless Linux server this normally means:

- desktop or virtual display, such as Xvfb/VNC/noVNC
- Linux QQ/NT QQ, or Windows QQ under Wine if Linux QQ cannot run mini-apps
- network access to QQ and `gate-obt.nqf.qq.com`

Without a real QQ mini-app runtime, the server cannot generate the farm code by itself.

## Operator flow

1. Start panel and capture services.
2. Start QQ on the server desktop.
3. User scans QQ login QR once.
4. Open QQ Classic Farm once so QQ downloads the mini-app source.
5. The watcher patches QQ Farm automatically. Manual command if needed:

```bash
node tools/watch-qq-farm-code-capture.js --capture-ws ws://127.0.0.1:9988/admin --username admin
```

6. Close and reopen QQ Classic Farm.
7. Watch `logs/code-capture.log` or `journalctl -u qq-farm-code-capture -f`.

Once captured, the bot side simulates the phone farm traffic from the server.

## Windows local flow

```powershell
powershell -ExecutionPolicy Bypass -File tools\start-local.ps1
```

Then log in to QQ on this machine, open QQ Classic Farm once, wait for `logs\patch.out.log` to show the patch, and reopen QQ Classic Farm. The captured code is sent to `/api/code-capture`, the account is created or updated, basic farm automation is enabled, and the worker starts.
